# Squall ŌĆö Slay the Spire 2 Mod

A Slay the Spire 2 character mod adding **Squall** (Final Fantasy VIII), built around the
**Crisis/Renzokuken finisher** system and the **Junction / Guardian Force (GF)** system.

This README documents the Junction/GF feature implementation (commit `claude's changes`),
what was built, and ŌĆö more importantly ŌĆö *why* it was built the way it was. The original
feature request is in `will's request.txt`.

---

## The Junction / GF feature ŌĆö overview

At the start of the first turn, the player chooses a Guardian Force to **Junction**
(shown as cards): Ifrit, Shiva, or Quezacoatl ŌĆö plus Leviathan and Diabolos if the
matching relics (`LeviathanScale`, `MagicLamp`) are owned. Junctioning applies the
matching passive power (`IfritPower`, `ShivaPower`, ŌĆ”), which also unlocks that GF for
summoning and magic generation.

Everything else in the feature keys off of *which GFs are currently junctioned*:

| Card | What it does |
|---|---|
| **Summon GF** (basic) | 8 (12) Block, Exhaust. Channel your junctioned GF; it is summoned in 2 turns. |
| **Perfect Junction** (ancient) | 1 cost, 8 (12) Block, Exhaust. Same, but summons **next** turn. |
| **Draw** (uncommon power, 2ŌåÆ1 cost) | At the start of each turn, add a basic Magic token (Fire/Blizzard/Thunder/Water/Demi) to your hand, based on the junctioned GF. |
| **Cast** (uncommon attack, 1 cost, Exhaust) | Immediately cast a powerful -aga Magic (Firaga/Blizzaga/Thundaga/Flood/Graviga) based on the junctioned GF. Upgrades with the card. |
| **Stock** (rare power, 2ŌåÆ1 cost) | Junction another GF mid-combat. |

Whenever more than one GF is junctioned, the player chooses between them on a
card-select screen; with exactly one, the choice is skipped.

Two new relics:

- **Solomon Ring** (rare) ŌĆö summoned GFs and the tokens generated by Draw are upgraded.
- **Junction Ring** (rare) ŌĆö junction one *additional* GF at the start of combat.

## Architecture: `GfRegistry` is the single source of truth

**File: `SquallCode/Mechanics/GF/GfRegistry.cs`**

The core design problem: five GFs ├Ś six consumers (first-turn junction, Summon GF,
Perfect Junction, Draw, Cast, Stock, plus the in-hand icon overlay) would naturally
explode into 30 hand-written `if (Owner.HasPower<IfritPower>()) ŌĆ” else if ŌĆ”` blocks ŌĆö
which is exactly what the first prototype of the first-turn junction in
`FirepowerRelicBase` looked like.

Instead, each GF is described once as a `GfEntry` record holding everything any consumer
needs:

- `IsJunctioned` / `ApplyJunctionPower` ŌĆö junction state and how to junction it
- `IsUnlocked` ŌĆö availability gate (always for the base three; relic-gated for
  Leviathan/Diabolos)
- `CreateGfCard` / `CreateDrawToken` / `CreateCastMagic` ŌĆö card factories for the summon
  card, the Draw token, and the Cast magic
- `GfCardTip` / `DrawTokenTip` / `CastMagicTip` ŌĆö hover-tip factories (upgraded variant
  selectable), used by the icon overlay
- `IconPath` ŌĆö the power icon shown on junction-related cards in hand

On top of that the registry provides shared flows so every consumer behaves identically:

- `Junctioned(player)` / `Junctionable(player)` ŌĆö the two queries everything filters by.
- `ChooseCard(...)` ŌĆö one pick-a-card UI. Uses the cinematic three-card choose screen
  when possible (it supports at most 3 cards) and falls back to a simple grid for 4ŌĆō5
  options. Skips the screen entirely for a single option.
- `JunctionNewGf(...)` ŌĆö the full "pick a GF you don't have and junction it" flow, used
  by the first-turn junction, Junction Ring, and Stock.
- `ChooseJunctionedGf(...)` ŌĆö "pick one of your junctioned GFs, displayed as *these*
  cards", used by Summon GF, Perfect Junction, Draw, and Cast. The card factory is a
  parameter because each consumer displays something different (GF card vs. token vs.
  magic).
- `HasSummonUpgrade(player)` ŌĆö the Solomon Ring check, kept in one place so the upgrade
  rule ("summons and Draw tokens, not Cast magic") can never drift between consumers.

**Why:** adding a sixth GF is now a single new `GfEntry` in one list ŌĆö every card, every
choice screen, and the icon overlay pick it up automatically.

## The GF summon cards

**Files: `SquallCode/Cards/Ancient/{Ifrit,Shiva,Quezacoatl,Leviathan,Diabolos}.cs`**

Previously empty stubs; now full attack cards (all 0-cost, Ancient rarity, marked
`IGFCard`):

| GF | Effect (upgraded) |
|---|---|
| Ifrit | 18 (22) damage to ALL enemies |
| Shiva | 12 (15) damage to ALL enemies, 12 (15) Block |
| Quezacoatl | 4 damage ├Ś 5 (6), random enemies |
| Leviathan | 15 (18) damage, 2 (3) Vulnerable, 2 (3) Weak |
| Diabolos | 10 damage + 20% (25%) of target's current HP |

Design notes:

- They are never in the player's deck ŌĆö they're created on the fly and **autoplayed**
  by `ChannelingGfPower` (or shown on selection screens). Their titles read
  "Junction Ifrit" etc. because the same cards double as the junction-choice UI.
- The `IGFCard` marker interface matters: `FirepowerRelicBase` and `FirepowerPower`
  both exclude `IGFCard` plays from building/consuming Firepower, so a summon doesn't
  eat the player's charged Firepower buff.
- Diabolos' %-HP hit uses `ValueProp.Unpowered` deliberately, so gravity damage doesn't
  scale with Strength/Firepower ŌĆö it's meant to be a fixed fraction, as in FF8.
  (This commit also fixed a compile error there: `CreatureCmd.Damage` takes
  `(ŌĆ”, CardModel? source, CardPlay? play)` as its trailing arguments, not
  `(ŌĆ”, Creature, CardModel)`.)

## `ChannelingGfPower` ŌĆö one buff for all five GFs

**File: `SquallCode/Powers/ChannelingGfPower.cs`**

The request was explicit: *"I don't want to create 5 different Summon GF buffs."*
So there is exactly one `ChannelingGfPower`, with the smart description
`"Summoning [gold]Junctioned[/gold] GF in [blue]{Amount}[/blue] turns."`

Implementation follows the base game's `TheBombPower` (the canonical
"tick down, then do a thing" power) closely:

- **`StackType.Counter`** ŌĆö `Amount` is the visible turns-remaining counter.
- **`InstanceType.Instanced`** ŌĆö playing two Summon GFs creates two *independent*
  countdowns instead of stacking one counter to 4 turns (same reason The Bomb is
  instanced).
- **Which GF to summon** is stored as an index into `GfRegistry.All`, kept in a
  `DynamicVar("GfIndex")` and set via `SetChosenGf(...)` right after the power is
  applied. This mirrors how `TheBombPower.SetDamage` smuggles extra state alongside
  `Amount`, and was chosen over ad-hoc internal data because DynamicVars are the
  game's sanctioned per-instance state container. If the index is ever missing, it
  falls back to the first junctioned GF rather than fizzling.
- **Tick timing:** `AfterPlayerTurnStart`. With amount 1 the GF is summoned at the start
  of the *next* turn (Perfect Junction's "summon NEXT turn"); with amount 2, the turn
  after (Summon GF). Summoning at the start of the player's turn (rather than turn end)
  reads more naturally with the countdown the player sees during their own turns.
- On trigger it creates the GF card fresh, upgrades it if Solomon Ring is held, and
  `CardCmd.AutoPlay`s it (the same autoplay mechanism Renzokuken already uses for
  finishers ŌĆö it safely handles "combat already over" and picks a random target for
  single-target GFs), then removes itself.
- The power's hover tip shows the actual GF card that will be summoned.

## Summon GF & Perfect Junction

**Files: `SquallCode/Cards/Basic/SummonGf.cs`, `SquallCode/Cards/Ancient/PerfectJunction.cs`**

Both follow the same flow: gain Block ŌåÆ choose a junctioned GF (screen skipped if only
one) ŌåÆ apply `ChannelingGfPower` ŌåÆ store the choice on the power instance.

- Summon GF: 2 cost, 8 Block, channels for **2** turns.
- Perfect Junction: 1 cost, 8 Block, channels for **1** turn (next turn).
- Both Exhaust. Upgrading raises Block 8 ŌåÆ 12 and nothing else, per the request ŌĆö
  the *summon itself* is upgraded by the Solomon Ring relic instead, so "upgraded GFs"
  stay relic-gated.
- The GF-choice screen shows the GF cards upgraded when Solomon Ring is held, so what
  the player sees is exactly what will be summoned.

## The 10 Magic token cards

**Files: `SquallCode/Cards/Token/*.cs`** (previously empty stubs)

All are 0-cost, `CardRarity.Token` (kept out of reward pools), and **Exhaust**.

Basic tokens (generated by Draw, upgraded by Solomon Ring):

| Token | GF | Effect (upgraded) |
|---|---|---|
| Fire | Ifrit | 3 (4) damage |
| Blizzard | Shiva | 2 (3) damage, 2 (3) Block |
| Thunder | Quezacoatl | 2 (3) damage ├Ś 2, random |
| Water | Leviathan | 2 (3) damage to ALL |
| Demi | Diabolos | 8% (10%) of target's current HP |

-aga magic (cast by Cast, upgraded by upgrading Cast):

| Magic | GF | Effect (upgraded) |
|---|---|---|
| Firaga | Ifrit | 15 (18) damage |
| Blizzaga | Shiva | 10 (12) damage to ALL, 10 (12) Block |
| Thundaga | Quezacoatl | 4 damage ├Ś 4 (5), random |
| Flood | Leviathan | 12 (16) damage to ALL, 1 (2) Weak to ALL |
| Graviga | Diabolos | 15% (20%) of current HP to ALL |

Design decisions (the spec left these open ŌĆö flagging for tuning):

- **Exhaust on every token.** Without it, autoplayed/unused tokens land in the discard
  pile and get reshuffled, silently growing the deck for the rest of combat (a Cast
  would permanently add a Firaga to the fight). Exhaust keeps tokens one-shot, matching
  Shiv-style tokens.
- **Basic-token upgrade values** weren't specified (Solomon Ring needs them to exist),
  so they got conservative +1s / +2% picks ŌĆö see the parenthesised numbers above.
- Demi/Graviga reuse Diabolos' unpowered %-HP damage pattern for the same
  "doesn't scale with Strength" reason.

## Draw, Cast, Stock

**Draw** (`SquallCode/Cards/Uncommon/DrawMagic.cs` + `SquallCode/Powers/DrawMagicPower.cs`)
was rewritten from an old draw-and-discard skill into what the request describes: a
2-cost (1 when upgraded) **power card**. The card itself just applies
`DrawMagicPower`; the power hooks `AfterPlayerTurnStart` and adds the junctioned GF's
token to the hand via `CardPileCmd.AddGeneratedCardToCombat` (choice screen when
multiple GFs are junctioned, Solomon-Ring upgrade applied at generation time).
The card/power split is required because power cards leave play ŌĆö something has to
live on the creature to fire every turn.

**Cast** (`SquallCode/Cards/Uncommon/CastMagic.cs`): 1-cost uncommon attack, Exhaust.
Chooses a junctioned GF (the choice screen shows the actual magic cards, upgraded if
Cast is upgraded), then `AutoPlay`s a freshly created magic token at the chosen target.
Its own `OnUpgrade` is intentionally empty ŌĆö upgrading Cast upgrades *what it casts*,
not the shell card. Uncommon was picked from the request's "rare/uncommon, you decide"
because the effect scales with junction count rather than being strong standalone.

**Stock** (`SquallCode/Cards/Rare/Stock.cs`): rare power card, 2 ŌåÆ 1 cost
(`EnergyCost.UpgradeBy(-1)`, the base game's cost-upgrade API). Its entire body is one
call to `GfRegistry.JunctionNewGf` ŌĆö the payoff of centralising that flow.

## Relics & the `FirepowerRelicBase` refactor

**`SolomonRing`** (`SquallCode/Relics/SolomonRing.cs`) is a passive marker; all logic
lives in `GfRegistry.HasSummonUpgrade(player)`. This is the "name TBA" relic from the
request that upgrades summons *and* Draw tokens ("Solomon Ring" is the FF8 item that
grants a GF, so the name fits the theme; trivial to rename later since it's referenced
in exactly one method).

**`JunctionRing`** (`SquallCode/Relics/JunctionRing.cs`) is likewise a marker. The
request allowed implementing it either in the relic or in `FirepowerRelicBase`; it went
into `FirepowerRelicBase`'s first-turn flow because that guarantees ordering (the extra
junction always happens immediately after the first one) instead of depending on relic
iteration order.

**`FirepowerRelicBase.AfterSideTurnStart`** previously contained ~45 lines of
hand-rolled junction logic: manually creating all five GF cards, five relic/power
checks to filter them, and five `if (selected == x) apply X` branches. That block is
now:

```csharp
await GfRegistry.JunctionNewGf(choiceContext, base.Owner);

//Junction Ring grants one additional Junction.
if (base.Owner.GetRelic<JunctionRing>() != null)
    await GfRegistry.JunctionNewGf(choiceContext, base.Owner);
```

Behavior is identical for the base case, plus it now shows upgraded GF cards when
Solomon Ring is held and handles the 4-and-5-option case gracefully (grid fallback).

## In-hand GF icons (`GfCardIconPatch`)

**File: `SquallCode/Mechanics/GF/GfCardIconPatch.cs`**

The request asked for the same treatment `CrisisCardPatch` gives Renzokuken (finisher
icons on the card in hand, with hover tips) applied to Summon GF, Perfect Junction,
Draw, Cast, and Stock. The new patch mirrors `CrisisCardPatch`'s structure exactly ŌĆö
Harmony postfixes on `NHandCardHolder._Ready` / `UpdateCard`, per-icon `Control`
containers inside the card hitbox, `NHoverTipSet` tips positioned left of the card ŌĆö
with two deliberate differences:

1. **Icons are built in code** (a `Control` + `TextureRect` loading the existing
   `*_power.png` GF textures from `GfEntry.IconPath`) instead of per-icon `.tscn`
   scenes. The crisis icons needed hand-authored scenes with baked resource UIDs;
   generating five more by hand adds Godot-editor churn for what is just
   "a texture in a box". Runtime construction keeps the whole feature in one file.
2. **Visibility and tips are data-driven** off the registry, per the request:
   - Stock shows GFs you *could* junction (`IsUnlocked && !IsJunctioned`);
     everything else shows GFs you *have* junctioned.
   - Hover tips: Draw ŌåÆ the token that will be generated; Cast ŌåÆ the magic that will
     be cast (upgraded if the Cast card is upgraded); Summon GF / Perfect Junction /
     Stock ŌåÆ the GF card itself (upgraded if Solomon Ring is held).

Icons stack down the right edge of the card (same anchor position as the crisis icons ŌĆö
no conflict, since no card is ever both Renzokuken and a junction card). Both patches
coexist because each only manages its own named containers.

  `SQUALL-SOLOMON_RING` / `SQUALL-JUNCTION_RING` relic entries were added.

## Known gaps / follow-ups

- **Card art**: none of the new cards (tokens, Cast, Stock) have portraits yet; they
  fall back to the placeholder `card.png` (logged, non-fatal).
- **Tuning knobs chosen without spec** (flagged above): basic-token upgrade values,
  Exhaust on tokens, Cast = Uncommon, both new relics = Rare, Solomon Ring's name.
- Tokens are ordinary Attacks, so they build Firepower progress like any other attack
  (only `IGFCard` summons and finishers are excluded). If that turns out too strong
  with Draw generating a free attack every turn, excluding tokens is a one-line change
  in `FirepowerRelicBase.IsValidAttack`.
- `ChannelingGfPower`'s chosen-GF index is combat-scoped state (like the base game's
  equivalent powers); a mid-combat save/reload would fall back to the first junctioned
  GF.

## Building

Requires Slay the Spire 2 installed (auto-discovered via Steam/registry, see
`Sts2PathDiscovery.props`) and the .NET 9 SDK:

```
dotnet build Squall.csproj
```

The build copies the DLL and manifest into the game's `mods/` folder automatically.
